# 集成日历功能

## Goal

为 Kairos 增加一个集成日历视图，让用户在同一日程视图中查看待办事项截止日期、课程表和考试时间，减少在“待办事项 / 课程表 / 考试倒计时”之间来回切换的成本。

## Product Positioning

Kairos 的核心差异化不是做一个更花哨的课程表，也不是复刻通用待办工具，而是面向学生真实学习节奏的“本地优先学业时间操作台”：

* 以学业时间为中心，而不是以任务列表为中心：课程、考试、待办、专注计时共同进入同一时间语境。
* 将计划与执行闭环：日历告诉用户什么时候该做什么，专注计时记录用户是否真的投入时间。
* 离线优先、数据自有：核心数据保存在本地，WebDAV 只作为用户可控同步方式，避免依赖平台账号或云服务。
* 跨桌面与移动：桌面适合规划和整理，移动端适合查看今天/本周安排与快速执行。
* 学生场景优先：周次、单双周、学期、考试倒计时、课程导入等是一级能力，而不是通用日历里的附属字段。

因此，集成日历的设计重点不是“显示更多事件”，而是把分散的学业信息整合成可执行的本周计划。

## What I Already Know

* 用户希望日历同时显示待办事项、课程表、考试截止时间。
* 用户担心新增日历后 5 个功能入口在移动端过于拥挤，需要先规划信息架构。
* 用户确认移动端底部第四个 Tab 使用“Kairos”，而不是“更多”或“我的”。
* 当前已有独立模块：
  * 待办事项：`Task.due_date` 是日期级字段，可表示截止日。
  * 课程表：`Course.day_of_week/start_time/end_time/week_pattern/semester_start_date` 表示按周重复的课程。
  * 考试：`Exam.exam_datetime/exam_end_datetime` 表示具体日期时间。
* 当前后端已有 `get_week_schedule`，可把课程和考试合并为一周时间轴。
* 当前 `WeekScheduleItem.kind` 只有 `"course" | "exam"`，还未支持 `"task"`。
* 当前主导航是番茄钟、待办事项、课程表、考试倒计时；新增日历需要增加导航入口。
* 当前前端是 React + TypeScript + Tailwind CSS v4 + Tauri IPC，离线优先，不应引入外部网络资源。
* 用户补充：日历需要支持寒暑假期间的日程，即没有正常课程周次时仍能按日期查看待办和考试。

## Assumptions

* 日历是新的顶层模块，而不是替换现有课程表模块。
* 第一版以“查看和跳转”为主，不在日历中实现完整编辑表单。
* 待办事项没有具体时间，默认作为全天/日期级事件显示。
* 已完成待办默认弱化显示或可过滤，避免干扰当前日程。
* 寒暑假日程第一版不新增独立“假期事件”数据表，优先复用待办截止日和考试时间。

## Requirements

* 提供一个集成日历入口，同时避免移动端底部导航过度拥挤。
* 移动端底部导航固定为 4 个主入口：专注、待办、日历、Kairos。
* 移动端不显示 Kairos 顶部 dock/header；品牌、设置、同步和次级功能统一收敛到 Kairos Tab。
* 移动端 Kairos Tab 承载课程表、考试倒计时、同步设置、主题切换等次级入口。
* 桌面侧边栏保留完整直达入口：专注、待办事项、日历、课程表、考试倒计时、同步。
* 第一版日历支持两种视图，默认进入日视图：
  * 周视图：采用课程表式周网格，包含时间轴与 7 天列；课程和考试按时间块定位，待办作为日期级截止事项显示在对应日期顶部。
  * 日视图：通过日期条切换日期，以课程卡片风格展示选中日期的综合日程，不使用时间轴。
* 日历必须显示三类事件：
  * 课程：按当前周和周次规则展开。
  * 考试：按考试开始/结束时间显示。
  * 待办：按 `due_date` 显示截止日。
* 考试在集成日历中以真实日期为准；即使 `semester` 为空或与当前课程学期不一致，只要考试日期落在当前日历周，也必须显示。
* 日历必须支持寒暑假日期周：
  * 前端按真实周起始日期浏览上一周/下一周，而不是只依赖学期第 N 周。
  * 当目标周没有有效学期锚点或没有课程时，仍显示该真实日期范围内的待办和考试。
  * 当目标周处在学期内且存在课程锚点时，课程继续按学期起始日推导周次并遵守单双周/周次规则。
* 三类事件必须有清晰视觉区分，包括类型、颜色或图标。
* 点击日历事件时，第一版跳转到对应原模块：
  * 待办事件跳转到待办事项。
  * 课程事件跳转到课程表。
  * 考试事件跳转到考试倒计时。
* 第一版不在日历内复制待办、课程、考试编辑表单。
* 支持移动端和桌面端：
  * 桌面端可浏览课程表式一周时间网格。
  * 移动端可在周视图和日视图之间切换；周视图使用横向滚动的课程表式网格，日视图保持无时间轴卡片列表。
* 加载、空状态、错误状态必须有明确 UI。
* 不破坏现有待办、课程、考试页面。

## Acceptance Criteria

* [ ] 主导航出现“日历”入口，并能切换到集成日历。
* [ ] 集成日历能加载并显示课程、考试和有截止日期的待办。
* [ ] 课程遵守已有周次规则，考试按具体日期展示，待办按截止日期展示。
* [ ] 寒暑假期间即使没有课程或学期周次锚点，也能按真实日期周显示待办和考试。
* [ ] 从学期周切换到假期周时，课程不会错误重复到假期日期中。
* [ ] 日历默认进入日视图，并提供“周视图 / 日视图”切换。
* [ ] 周视图可按课程表周视图方式浏览一周内的综合日程，课程和考试按时间块定位，待办显示为日期级事项。
* [ ] 日视图可通过日期条切换日期，并以课程卡片风格查看选中日期的综合日程，内容不重叠。
* [ ] 三类事件有明显区分，长文本不会撑破布局。
* [ ] 点击课程、考试、待办事件时能导航到对应现有模块。
* [ ] Tauri 不可用或加载失败时显示错误态。
* [ ] 通过 `npm run lint`、`npx tsc --noEmit`，后端变更通过相关 Rust 测试。

## Out of Scope

* 第一版不做外部日历导入/导出，如 ICS、CalDAV。
* 第一版不做拖拽改期。
* 第一版不做周期性待办。
* 第一版不新增独立假期计划数据模型；假期安排通过带截止日期的待办和已有考试时间进入日历。
* 第一版不做日历内完整编辑表单；点击事件可先跳转/打开现有模块或详情弹层。
* 第一版不做跨设备实时同步逻辑，仍复用现有离线数据和后续 WebDAV 同步。

## Navigation Planning

### Problem

当前底部导航已有 4 个入口：番茄钟、待办事项、课程表、考试倒计时。新增日历后变成 5 个入口，移动端会出现文字拥挤、触控目标变窄、主功能优先级不清的问题。

### Options

#### Option A: 日历作为主入口，课程/考试作为二级视图（推荐）

底部导航保留 4 个入口：

* 专注
* 待办
* 日历
* Kairos

日历内部提供筛选或 Tab：全部、课程、考试、待办。

优点：移动端最清晰，日历承担“时间管理总览”，课程和考试仍可通过日历或 Kairos 入口进入。
缺点：需要调整现有“课程表 / 考试倒计时”的入口层级。

#### Option B: 保留 5 个底部入口

底部导航直接显示：专注、待办、日历、课程、考试。

优点：所有模块直达。
缺点：移动端拥挤，标签更短也会降低辨识度，不利于长期扩展。

#### Option C: 桌面完整入口，移动端 4 个入口 + Kairos

桌面侧边栏保留全部入口；移动端底部只保留高频入口，其余放入 Kairos。

优点：兼顾桌面直达和移动端简洁。
缺点：两端导航结构不完全一致，需要维护清晰映射。

### Recommended Navigation Direction

采用 Option C，第一版移动端底部保留：

* 专注
* 待办
* 日历
* Kairos

桌面侧边栏可以显示：

* 专注
* 待办事项
* 日历
* 课程表
* 考试倒计时
* 同步

移动端 Kairos 中放课程表、考试倒计时、同步设置和主题切换。这样既解决 5 入口拥挤，又不牺牲桌面效率。

### Decision

已确认采用 Option C：移动端底部导航为“专注 / 待办 / 日历 / Kairos”，课程表和考试倒计时降级到移动端 Kairos Tab；桌面端侧边栏保留所有功能直达。Kairos Tab 同时承担品牌露出、设置、同步和更多功能入口。

## Feasible Approaches

### Approach A: 独立集成日历模块（推荐）

新增 `CalendarView` 组件和 `get_calendar_week` 命令，后端聚合课程、考试、待办为统一事件模型。

优点：边界清晰，后续支持月视图、日视图、过滤器更自然；不会污染课程表模块。
缺点：需要新增一组类型和后端命令。

### Approach B: 扩展现有课程表周视图

直接把待办加入现有 `get_week_schedule` 和 `CourseSchedule`，课程表变成综合周视图。

优点：改动量较小，可复用大量 UI。
缺点：课程表职责变宽，后续维护容易混乱；用户想看纯课程表时会被待办干扰。

### Approach C: 前端聚合三类列表

前端分别调用 `get_all_tasks`、`get_week_schedule`、`get_all_exams` 后合并展示。

优点：后端改动少。
缺点：日期/时区/周次逻辑分散到前端，重复已有后端时间规则，后续更容易出现不一致。

## Recommended Direction

采用 Approach A：新增独立集成日历模块，后端提供统一聚合命令，前端只负责渲染和交互。

## Vacation Schedule Decision

第一版采用“日期周优先”的假期支持：集成日历请求后端时传入真实周起始日期，后端以该日期范围筛选待办和考试；如果存在学期起始日，则用真实周起始日期反推课程周次并展开符合规则的课程。这样寒暑假期间不会因为缺少课程或学期锚点而空白，也不会把课程表模块改造成假期计划模块。

## View Decision

第一版默认进入日视图，并提供周视图切换。日视图通过日期条选择日期，并以课程卡片风格展示单日综合日程，不使用时间轴；周视图采用类似课程表的时间轴网格：横向为周一到周日，纵向为上课/考试时间，课程和考试定位为时间块，待办因没有具体时间而显示在对应日期顶部的截止事项区域。月视图不进入 MVP。

## Interaction Decision

第一版点击日历事件时跳转到原模块，不在日历内直接编辑。这样避免重复维护三套表单逻辑，保持日历作为综合总览入口。
